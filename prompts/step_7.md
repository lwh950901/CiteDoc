## 任务拆解

### 1. 多轮对话支持

#### 1.1 后端：维护对话历史并传入 Prompt

**文件**: `app/api/chat/route.ts`

**改动点**：
- 请求体新增可选字段 `history`：`Array<{ role: 'user' | 'assistant'; content: string }>`
- 在构建 LLM 调用时，将历史消息拼接到 messages 数组中，放在系统提示词之后、当前用户问题之前
- 仅保留最近 N 轮（例如 5 轮，即 10 条消息），防止 token 超限
- 系统提示词中增加：“如果用户的问题是对上一轮回答的追问，请结合对话历史理解上下文。”

**改造示意**：
```ts
// 在 POST 函数中
const { documentId, question, history } = await req.json();

// 截取最近 10 条历史消息
const recentHistory = (history || []).slice(-10);

const messages = [
  { role: 'system', content: systemPrompt },
  ...recentHistory,
  { role: 'user', content: userMessage },
];

const completion = await llmClient.chat.completions.create({
  model: process.env.LLM_MODEL || 'deepseek-v4-flash',
  messages,
  temperature: 0.3,
});
```

> **注意**：历史消息中不附带检索到的文档片段，仅保留用户问题和 AI 回答的纯文本。每次请求时重新检索文档片段作为新鲜上下文，确保后续追问也能获取相关原文。

#### 1.2 前端：在 ChatPanel 中维护消息数组

**文件**: `components/ChatPanel.tsx`

**改动点**：
- 将 `answers` 状态改为 `messages: Array<{ role: 'user' | 'assistant'; content: string }>`
- 每次用户发送问题时，追加一条 `{ role: 'user', content: question }`，发送 API 请求时把之前的所有消息作为 `history` 传入
- AI 回答流式构建完成后，追加一条 `{ role: 'assistant', content: fullAnswer }`
- 渲染对话气泡：用户消息右对齐（蓝色背景），AI 消息左对齐（灰色背景），AI 消息中的角标保持可点击
- 如果用户在一个回答还在流式输出时发送新问题，需要中断当前请求（AbortController），然后发起新请求

### 2. 全流程错误处理与状态覆盖

#### 2.1 后端错误处理增强

**涉及文件**：
- `app/api/upload/route.ts`
- `app/api/chat/route.ts`
- `app/api/documents/[id]/embed/route.ts`

**统一规范**：
- 所有 API 路由使用 `try-catch`，捕获后返回结构化 JSON 错误：
  ```json
  { "error": "人类可读的错误描述" }
  ```
- 状态码规范：
  - `400`：参数缺失或格式错误
  - `404`：文档不存在
  - `413`：文件过大
  - `429`：API 调用频率限制（来自 LLM/Embedding 服务）
  - `500`：服务器内部错误
- 对 LLM 调用和 Embedding 调用单独捕获，区分“服务商报错”和“网络超时”：
  - 网络超时：返回 504，提示“AI 服务响应超时，请稍后重试”
  - 密钥无效：返回 500，提示“AI 服务配置错误，请联系管理员”
- 流式接口中的错误：如果 LLM 调用本身失败，返回一个包含错误信息的 SSE 事件，让前端能展示错误而非卡在 loading：
  ```ts
  controller.enqueue(encoder.encode(`event: error\ndata: ${JSON.stringify({ error: 'xxx' })}\n\n`));
  ```

#### 2.2 前端状态管理规范化

**涉及文件**：
- `components/FileUpload.tsx`
- `components/ChatPanel.tsx`
- `components/DocumentViewer.tsx`

**每种组件必须覆盖的状态**：

| 状态 | 表现 |
|------|------|
| **idle** | 正常初始状态，引导用户操作 |
| **loading** | 骨架屏或转圈动画，操作按钮禁用 |
| **empty** | 明确提示“暂无数据，请先上传文档”或“当前文档没有内容” |
| **error** | 红色提示条，显示具体错误信息，附带“重试”按钮 |
| **streaming** | 打字光标闪烁，中断按钮可见 |
| **no-result** | 检索无结果时显示“文档中未找到相关信息，请尝试换一种问法” |

**关键交互保护**：
- 上传中禁止重复提交（disabled 按钮 + loading 遮罩）
- 流式输出中禁止发送新消息（或允许但弹出确认“当前回答正在生成，确定要中断吗？”）
- 文档未向量化时提问，前端提示“文档正在处理中，请稍后再问”（可查 GET `/api/documents/:id/embed` 进度）

#### 2.3 特定错误场景处理清单

- [ ] 上传非 PDF/Word 文件 → 前端 toast 提示“仅支持 PDF 和 Word 文件”
- [ ] 上传文件 > 10MB → 提示“文件大小不能超过 10MB”
- [ ] PDF 加密或损坏 → 提示“文档解析失败，请确认文件未加密或损坏”
- [ ] SiliconFlow API Key 无效 → 向量化或检索时返回明确错误
- [ ] DeepSeek API Key 无效 → 问答时返回明确错误
- [ ] 网络断开 → 前端 fetch 捕获 TypeError，提示“网络连接异常，请检查网络”
- [ ] 用户中断流式回答 → AbortController 取消，前端正常恢复输入
- [ ] 原文太长（如 > 500KB）→ DocumentViewer 做虚拟滚动或分段懒加载（MVP 可先加一个最大显示长度截断 + “查看完整文档”按钮）

### 3. 用户体验打磨

#### 3.1 上传流程优化
- 上传成功后自动跳转到该文档的问答页（路由跳转 `router.push(\`/${docId}\`)`）
- 上传后展示向量化进度条（轮询 GET `/api/documents/:id/embed`），显示“正在处理文档... 3/15 块已向量化”
- 向量化完成前，“开始提问”按钮为 disabled 状态并提示等待

#### 3.2 问答交互打磨
- 输入框支持 Enter 发送，Shift+Enter 换行
- 发送后自动清空输入框
- 对话区域自动滚动到最新消息（`useEffect` + `scrollIntoView`）
- 角标点击后的高亮效果：持续 3 秒后自动淡出（用 `setTimeout` 清除 `activeChunkId` 状态）
- 点击角标时如果原文在该段落，滚动平滑过渡；如果原文未加载对应 chunk，静默（已有 chunk 渲染，不会出现此情况）

#### 3.3 移动端适配
- 双栏在 `<768px` 宽度时自动切换为上下布局
- 对话区在顶部，原文区在底部或通过 Tab 切换
- 角标点击时，移动端自动切换到原文 Tab 并滚动（可选：用一个浮动的“查看原文”按钮替代自动跳转）
- 触摸友好：按钮最小 44x44px，文本不溢出

#### 3.4 空状态引导
- 首页（无文档时）显示一个空状态插画/图标 + “上传你的第一份文档开始提问”
- 文档列表为空时同样提示

### 4. 异常边界测试用例（供手动验证）

| 操作场景 | 预期结果 |
|---------|----------|
| 直接访问 `/chat` 而不上传文档 | 提示“请先上传文档” |
| 快速连续点击“发送”按钮 | 第二次点击无效或被中断提示拦截 |

| 提问一个完全超出文档范围的问题 | 返回“根据现有资料无法回答”，无角标 |
| 点击角标后立即点击另一个角标 | 前一个高亮消失，新段落高亮并滚动 |
| 浏览器窗口缩放 | 双栏比例正常，移动端自动切换布局 |
